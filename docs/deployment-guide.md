# Deployment Guide

## Scope

This guide covers local and self-hosted deployments for a trusted LAN or single host setup.

Use `uv` plus `pnpm` as the primary workflow. Keep Docker available as a fallback when host-side Python or Node setup is not practical.

This project does not target cloud deployment, public internet exposure, multi-user hosting, or CI orchestration in this guide.

## Recommended paths

| Path | Status | What it is for | Notes |
| --- | --- | --- | --- |
| Linux + CUDA | Primary | Main local and self-hosted path on a Linux GPU host | Full project workflow target |
| WSL + ROCm | Supported target | Full-function path inside WSL when the runtime reports `wsl-rocm` | Treat WSL as the host environment |
| pip compatibility | Compatibility path | Backend dependency install path for Python 3.11 environments that cannot use `uv` directly | Use for backend packaging compatibility, not as the preferred full stack workflow |
| Docker fallback | Fallback only | Use when you need containerized API, worker, and web services | Do not treat Docker as the default path |

Unsupported or out of scope for this document:

- native Windows GPU deployment
- native macOS GPU deployment
- public internet exposure

## Shared requirements

### Required tools for the primary workflow

- Python `3.11`
- `uv`
- Node.js `>=20`
- `pnpm@10.33.2` or newer pnpm 10 release
- host media tools on `PATH`: `BBDown`, `yt-dlp`, `ffmpeg`, `ffprobe`

The backend dependency source of truth is `backend/pyproject.toml`. The checked-in `backend/requirements.txt` is generated by `./scripts/export_backend_requirements.sh` and should not be edited by hand.

### Shared setup

From the repo root:

```bash
uv sync --project backend --frozen
pnpm --dir web install --frozen-lockfile
```

The local scripts create repo-local data and cache directories automatically. By default they use:

- `APP_DATA_DIR=./data`
- `APP_MODEL_CACHE_ROOT=./data/model-cache`
- `APP_WHISPERX_MODEL_CACHE_DIR=./data/model-cache/whisperx`
- `HF_HOME=./data/model-cache/hf`
- `TRANSFORMERS_CACHE=./data/model-cache/hf`

## Linux + CUDA primary path

Use this path for the normal local or self-hosted deployment on a Linux NVIDIA host.

### Start the full stack

```bash
./scripts/dev_up.sh
```

That script:

- starts the API with `./scripts/dev_api.sh`
- waits for `http://127.0.0.1:8000/api/health`
- starts the worker with `./scripts/dev_worker.sh`
- starts the web UI with `./scripts/dev_web.sh`

Default local endpoints:

- API: `http://127.0.0.1:8000`
- Web UI: `http://127.0.0.1:5173`

### Run the services in separate terminals

Terminal 1:

```bash
./scripts/dev_api.sh
```

Terminal 2:

```bash
./scripts/dev_worker.sh
```

Terminal 3:

```bash
./scripts/dev_web.sh
```

Useful environment overrides:

- `APP_HOST` and `APP_PORT` for API bind address and port, default `0.0.0.0:8000`
- `VITE_HOST` and `VITE_PORT` for the web dev server, default `0.0.0.0:5173`
- `VITE_API_BASE_URL`, default `http://127.0.0.1:8000/api`
- `APP_DATA_DIR` if you want storage outside `./data`
- `APP_BILIBILI_COOKIE_PATH` if downloads require a cookie file

For LAN browser access from another machine, set `VITE_API_BASE_URL` to the host LAN address before starting the web server, for example:

```bash
VITE_API_BASE_URL=http://192.168.1.50:8000/api ./scripts/dev_web.sh
```

## WSL + ROCm full-function target

Use this path when you run the project inside WSL and the backend runtime reports ROCm through PyTorch.

This is a full-function target, not a reduced feature mode, as long as the runtime profile resolves to `wsl-rocm` and the required host media tools are available inside the WSL environment.

### Start the stack inside WSL

Run the same commands from your WSL shell:

```bash
uv sync --project backend --frozen
pnpm --dir web install --frozen-lockfile
./scripts/dev_up.sh
```

Or split the processes across terminals:

```bash
./scripts/dev_api.sh
./scripts/dev_worker.sh
./scripts/dev_web.sh
```

### What to verify

After startup, check the runtime profile:

```bash
python3 - <<'PY'
import json
import urllib.request

with urllib.request.urlopen('http://127.0.0.1:8000/api/runtime/capabilities', timeout=5) as response:
    payload = json.loads(response.read().decode('utf-8'))

print(json.dumps(payload, indent=2, ensure_ascii=False))
PY
```

For a healthy ROCm-backed WSL setup, the payload should report:

- `detected_profile: "wsl-rocm"`
- `accelerator.backend: "rocm"`
- `accelerator.available: true`

If the API reports `cpu-only` instead, startup still succeeds, but the runtime has fallen back to CPU mode and you should treat the environment as degraded until the ROCm path is fixed.

## pip compatibility path

Use this only when you need backend compatibility with a plain `pip` environment. The preferred full stack workflow remains `uv` plus `pnpm`.

### Regenerate and verify the backend requirements file

```bash
./scripts/export_backend_requirements.sh --check
```

If you need to refresh it:

```bash
./scripts/export_backend_requirements.sh
```

### Install backend dependencies with pip

Use Python 3.11 from the repo root:

```bash
python3.11 -m venv .venv-pip
. .venv-pip/bin/activate
python -m pip install --upgrade pip
python -m pip install -r backend/requirements.txt
```

Important limits for this path:

- treat it as a backend compatibility install path
- keep `backend/pyproject.toml` as the canonical dependency source
- do not assume it replaces the `uv` workflow for the full project lifecycle
- do not assume it guarantees full WSL plus ROCm parity by itself

If you use the pip path and still want the web UI, install frontend dependencies separately with:

```bash
pnpm --dir web install --frozen-lockfile
```

## Docker fallback

Use Docker only when the host-side `uv` plus `pnpm` workflow is not practical.

### Start the fallback stack

```bash
docker compose -f infra/docker-compose.yml up -d --build api worker web
```

To stop it:

```bash
docker compose -f infra/docker-compose.yml down
```

The fallback stack still serves:

- API on port `8000`
- Web UI on port `5173`
- Worker as the same single-worker loop

If browser clients connect from another LAN device, set `VITE_API_BASE_URL` before startup so the browser points at the host API address:

```bash
VITE_API_BASE_URL=http://192.168.1.50:8000/api docker compose -f infra/docker-compose.yml up -d --build api worker web
```

Use the Docker smoke path only as a fallback verification route:

```bash
./scripts/release_smoke_docker.sh
```

## Capability checks and warning semantics

Runtime capability checks are visibility features. They do not block startup.

What surfaces where:

- API readiness stays available at `/api/health`
- `/api/health` adds `runtime_capabilities` with `status`, `detected_profile`, and `warnings`
- full capability details are exposed at `/api/runtime/capabilities`
- the API writes one startup log line on logger `app.runtime` with event `runtime_capabilities_startup`
- worker stage logs append `worker_preflight_warning=<json>` when warnings exist
- the web UI shows runtime status so operators can see warnings without reading raw logs

The full capability payload uses these top-level keys:

- `status`
- `detected_profile`
- `platform`
- `accelerator`
- `dependencies`
- `warnings`

Warning semantics:

- missing GPU runtime reports a warning such as CPU-only mode
- missing host tools such as `BBDown`, `yt-dlp`, `ffmpeg`, or `ffprobe` report warnings
- missing optional diarization support reports a warning
- warnings stay visible in API responses, UI, and logs
- warnings do not stop API or worker startup

If core Python dependencies such as `torch`, `transformers`, or `whisperx` are missing, the capability payload can report `status: "error"`. That status is still observational. It does not turn the startup sequence into a hard block, but the system is not ready for real processing until the missing dependency is installed.

Primary local path:

Docker fallback path:

Backend requirements artifact:

```bash
python3 - <<'PY'
import json

with urllib.request.urlopen('http://127.0.0.1:8000/api/health', timeout=5) as response:
    payload = json.loads(response.read().decode('utf-8'))

PY
```

## Troubleshooting

### `./scripts/dev_up.sh` fails before the stack is ready

Check that these commands exist on the host:

```bash
python3 - <<'PY'
import json
import shutil

print(json.dumps({
    'uv': shutil.which('uv'),
    'pnpm': shutil.which('pnpm'),
    'python3': shutil.which('python3'),
    'BBDown': shutil.which('BBDown'),
    'yt-dlp': shutil.which('yt-dlp'),
    'ffmpeg': shutil.which('ffmpeg'),
    'ffprobe': shutil.which('ffprobe'),
}, indent=2, ensure_ascii=False))
PY
```

### API starts, but capability status shows `cpu-only`

- on Linux, confirm the CUDA runtime is visible to PyTorch
- on WSL, confirm the runtime reports `wsl-rocm`
- read `/api/runtime/capabilities` and the API startup log before running a real task

### The worker starts, but a stage log contains `worker_preflight_warning=`

That warning is non-blocking. It means the worker is running, but the runtime detected a degraded condition that should be fixed before trusting performance or output quality.

### The pip compatibility install fails

- confirm you are using Python `3.11`
- rerun `./scripts/export_backend_requirements.sh --check`
- install from the repo root so `./backend` resolves correctly from `backend/requirements.txt`

### Docker fallback cannot reach the API from another LAN device

Set `VITE_API_BASE_URL` to the host LAN address before `docker compose up`, then reload the web UI from the client machine.
